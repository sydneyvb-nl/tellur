# Aikido SAST and dependency triage — 2026-07-13

This record maps the supplied Aikido report to the actual Tellur trust
boundaries. A scanner finding is treated as a lead, not automatically as proof
of exploitability.

## Remediated findings

| Category | Assessment | Remediation |
| --- | --- | --- |
| DOM XSS in `web/index.html` | Confirmed. Local and hub API fields reached `innerHTML` template strings. A malicious imported event or hub value could create markup or an event handler in the browser. | All data-driven rendering now uses `textContent`, `createElement`, and `replaceChildren`; percentages are converted to finite values and clamped before CSS assignment. A regression test rejects `innerHTML`, `outerHTML`, and `document.write` in the embedded dashboard. |
| Dynamic SQLite identifiers in `storage/chain.rs` | Not externally exploitable in the reported revision: every caller supplied internal constants and values were bound. The public string-shaped helper plus release-disabled `debug_assert!` was nevertheless an unsafe future boundary. | `HeadRef` is now a closed `Audit`/`Event` enum and both queries are static SQL. Callers can only supply a bound key value. |
| Mutable third-party GitHub Action references | Confirmed supply-chain risk. | Every reported third-party action is pinned to a full commit SHA while retaining its prior major-version comment. |
| Broad release workflow token | Confirmed least-privilege issue. | Workflow default is `permissions: {}`; build/editor jobs receive `contents: read`; only the release job receives `contents: write`. |
| `smallvec` 1.15.1 | The vulnerable optional feature was not enabled in Tellur's resolved graph, but the affected version was present. | Lockfile updated to 1.15.2. |
| `mio` 0.8.11 | Present through `notify` 6 on Windows-capable builds; the vulnerable named-pipe API is not called directly by Tellur. | `notify` updated to stable 8.2.0, removing `mio` 0.8.11; the graph now resolves `mio` 1.2.1 only. |

## Path traversal findings: not applicable

The 15 reported filesystem reads do not cross a restricted-directory security
boundary:

- adapter reads (`antigravity`, `claude_code`, `codex`, `copilot`, `cursor`,
  `gemini`, `generic`, and shared `import`) receive the path explicitly supplied
  by the logged-in OS user to `tellur import`; absolute paths are supported
  intentionally;
- Claude settings and setup reads are derived from that same user's home/config
  directories, not from network input;
- policy/report, daemon-token, and event-log paths are derived from a discovered
  `RepoStorage` root and fixed `.tellur` subdirectories;
- the file watcher consumes paths emitted by Git for the repository being
  watched. Its reported read computes only an untracked file's line count; it
  does not return file contents to a remote caller.

The local CLI/daemon trust boundary is the logged-in OS user. Reading a path the
same user explicitly selected is not privilege escalation or sandbox escape.
Forcing these APIs under the current repository root would break supported
imports from global agent transcript/config locations without closing an
attacker-controlled boundary. Network-facing source-file access is a separate
Team Hub blob-proxy boundary and already uses provider allowlists, HTTPS-only
URLs, and server-side path/template validation.

## Security invariants

- Untrusted dashboard values never enter an HTML parser.
- SQL values are bound parameters and SQL identifiers cannot be supplied by a
  caller.
- Third-party workflow code is immutable for a reviewed commit.
- Write permission exists only in the release job that publishes assets.
- The resolved dependency graph contains neither `smallvec` 1.15.1 nor `mio`
  0.8.11.

Verification commands and final test totals are recorded in
[`PROJECT_STATUS.md`](../../PROJECT_STATUS.md).

The original DOM exploit was also replayed through a real `tellur daemon` and
headless Chromium: a file event containing `<img id=xss-probe ... onerror=...>`
rendered as literal text, created zero `#xss-probe` elements, and did not execute
the handler.
